What is HTML5?

Unfortunately it's not simple

So a bit of history is required

1991

History of HTML5

The Start of the Web

Aug 6th, 1991: From CERN, Tim Berners-Lee releases WorldWideWeb on the wide world.

Technically we should start much earlier, but that's when things get interesting (for us).

Must have been a good year: Linus Torvalds announced Linux 19 days later.

WorldWideWeb - the original browser

We've come a long way!

1993-1997

History of HTML5

Rise of the Web as a Platform

Feb 1993: Mosaic browser released; first truely graphical browser

Mar 1993 First commercial use of the Internet permitted

Sep 1994 W3C formed to steward web technologies and standards

1995 Bill Gates sees Web as a platform threat to Windows and pivots the company to focus on the Internet.

Microsoft "intends to cut off Netscape's air supply." The original browser wars begin.

Dec 1995 Javascript ships with Netscape 2.0

1996 CSS 1: some parts ship in IE 3.0

1997 HTML4 "standard" released

1997-2007

History of HTML5

Fall of the Web as a Platform

For the next 10 years, the web stagnates as a platform.

Oct 1997: IE 4.0 bundled with Windows for first time

May 1998: civil suite filed against Microsoft primarily related to browser/OS integration.

1998: Netscape Navigator becomes open sourced as Mozilla.

Oct 2001: IE 6.0 released

2002: IE achives over 90% marketshare.

IE stays above 60% until 2009.

W3C is stuck in an XML/XHTML morass.

But the Web moves ahead in spite of browser stagnation by repurposing old functonality and growing on the server-side.

2004-Present

History of HTML5

Rise of the Web as a Platform, Part 2

2004: Mozilla and Opera jointly propose to the W3C that it refocus on HTML (instead of XHTML). Proposal fails.

In response, Mozilla and Opera form the WHATWG (Web Hypertext Application Working Group) to start work on HTML5.

Jun 2007: iPhone released. Innaguarates the mobile web.

Notably: no Flash.

2007: W3C comes around. Adopts WHATWG's HTML5 as a starting point.

Jan 2008: W3C releases first HTML5 working draft.

Sep 2008: Google releases Chrome on the world. Browser is reframed once again as a platform instead of a product.

2009: W3C quietly dumps XHTML 2.0 and focuses full energy on HTML5 (the cancer is excised). HTML5 explodes into the community consciousness.

Apr 2010: Steve Jobs issues "Thoughts on Flash" manifesto in which he concludes that Flash is no longer necessary due to HTML5.

June 2011: Microsoft demos Windows 8 Metro interface. HTML/Javascript/CSS is positioned as primary way to build new Windows applications.

The Static Web

W3C HTML5 - Web Workers

Web Workers

Browsers are multi-threaded and/or multi-process.

Each Javascript context is single threaded.

Javascript/DOM APIs that might block are asynchronous with callbacks.

Moore's Law continues at full speed but in the form of CPU core count rather than clock frequency.

This is a problem for single threaded applications.

Workers enable Javascript multiprocessing. Use message passing to avoid concurency issues.

Workers cannot see the DOM.

Examples:
  • Ray Tracer
  • Letter Swarm

"Native" App vs Web App

What's the difference?

"Native" App vs Web App

Compiled Object Code vs Interpreted Source?

Nope.

Python/perl/ruby programs are native but interpreted.

Andriod apps are Dalvik bytecode; somewhere in the middle.

Javascript is a compile target for many languages.

Google's NaCl projects allows compiled code in web apps

"Native" App vs Web App

Installed vs Loaded On Demand?

Nope.

Web Apps can work offline.

Some native apps update (parts of) themselves on startup.

"Native" App vs Web App

Run in browser or not?

Nope.

Web apps can run standalone or from shortcuts (with no browser UI)

"Native" App vs Web App

Integration with Native APIs?

Nope.

Phonegap/Cordova blurs the line

Apps on Windows Metro, webOS, Tizen, QT7: native or web apps?

"Native" App vs Web App

App Marketplace vs Web Download?

Nope.

Chrome Web Store is for web apps.

Mozilla Marketplace will be generic web app marketplace.

PhoneGap/Cordova can package up web apps for Android/Apple app markets

"Native" App vs Web App

So What is the Difference?

Really comes down to control of distribution, discovery and monetization.

iOS apps are in Apple's walled garden

  • Pros: high signal to noise, easier to monetize.
  • Cons: Apple takes a cut, beholden to Apple's whims.

"Native" apps are products, web apps more like services (kind-of).

Web apps are harder to sell as a product, but easier to monetize via advertising.

Richness vs Reach

"Native" Apps Have Richness

"Native" apps can maximize the richness of the user experience.

Able to take full advantage of the hardware.

Target hardware and software are finite, knowable and testable.

Least common denominator factor is less of an issue.

Web Apps Have Reach

Number of people/platforms that can be reached is maximized

Smartphones will always have web browsers, therefore "Native" app reach will always be a subset of web app reach.

Limitless variations possible of hardware and browser versions

Do you go after "least common denominator" or sacrifice your reach.

HTML5 Application Fitness

Good Match

Content and media delivery (HTML5 has become one of the best design platforms)

Casual games (non-twitch)

Apps with tight social network integration

Not Yet A Good Match

Device specific software/utilities: camera, contacts, etc

Twitch games

Mostly offline apps (soon but not yet)

Sophisticated animation/interaction requirements (however the line is moving quickly)

Consider Progressive Enhancement

  1. Develop Web App First

  2. Use PhoneGap/Cordova or AppMobi to enter "native" marketplaces

  3. When you encounter richness barriers, convert sensitive parts to "native" languages, maintain core design/content rendering in a UIWebView (or equivalent)

  4. If necessary, convert whole app to the "native" language.

Final Thoughts

  • The Web is improving extremely rapidly. The next 1-2 years will see the gap between web apps and "native" marketplace delivered apps narrow substantially.
  • Adobe has been bitten by the HTML5 bug and is now a major contributor. Look for excellent HTML5 content production and workflow tools to arrive in force.
  • On the other hand, watch out for Apple. Initially a key driver of the Web, they may begin to consider web apps a threat to their marketplace.
  • Android is currently a problem. Even the latest Android version lags in web features. However, Chrome for Android is coming soon. Prossibly as default in next major release.
  • In app purchases can be much more lucrative than initial app sale. Consider the case of TF2. There are several ways of enabling in-app purchases with web apps.
    • PhoneGap/Cordova have some native in app purchase options.
    • AppMobi claims to have seamless native in app purchase.
    • Chrome Web Store has in app purchases.
    • Mozilla Marketplace has in-app purchase capability.
  • Keep an eye on Boot to Gecko and Chrome OS
